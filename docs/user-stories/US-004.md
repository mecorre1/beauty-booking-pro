## Context

## Story
As a client, I want to enter my contact details and confirm so that I have a reserved appointment.

## Requirements
- Add `Booking`, `Salon`, and relationships per SPEC (`slot_id`, `service_id`, snapshots: `price_at_booking`, `salon_address_at_booking`, `location`, optional `home_address`, client fields, `status`, unique `edit_token`).
- **Public:** `POST /api/public/bookings` (body: slot id, service id, location, home address if needed, client fields). Validate slot `is_available`, slot window vs. service duration, **one booking per slot**, **no double booking**.
- **At-home buffer (SPEC):** When `location=home`, treat **internal** buffer after `end_time` as occupied for overlap checks with other bookings (even if buffer is not shown to clients).
- On success: set slot unavailable (or equivalent), `status=confirmed`, persist snapshots from current `Salon` row and computed price.
- **Confirmation email:** SPEC requires it — implement a small `EmailSender` protocol + default no-op or in-memory test double; in dev log/print; pytest asserts the app **attempted** send with subject/body containing edit link placeholder. (No real SMTP required for MVP unless chosen.)
- **Frontend:** New confirm view: form name/email/phone; submit; show success with **edit/cancel** link using `edit_token` path (US-005/US-006 flesh out).

## Acceptance Criteria
- [x] Pytest: happy path 201, booking row + snapshots; slot `is_available` false.
- [x] Pytest: double booking same slot → 409 (or documented code).
- [x] Pytest: home booking blocks overlapping slot via buffer semantics.
- [x] Vitest: confirm view validation + success state.

## Out of scope
- Full client edit/cancel flows (US-005/006); admin views.
